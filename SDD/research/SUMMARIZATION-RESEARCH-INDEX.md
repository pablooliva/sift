## Research Areas Detail

### Area 1: Error Handling Patterns

**Why it matters:** ML models fail unpredictably; web applications must handle failures gracefully

**Key recommendations:**
1. Use 60-second timeout for text summarization (DistilBART)
2. Implement exponential backoff for retries (2^attempt seconds)
3. Always return fallback content instead of showing error to user
4. Track error codes to distinguish retryable vs. permanent failures

### Area 3: Performance Optimization

**Why it matters:** Summarization adds processing time; must not block user workflows

**Key recommendations:**
1. Use async processing for documents with summaries (threading for MVP)
2. Queue-based processing with persistence handles crashes
3. Dynamic timeout: `base_timeout * (text_length / expected_length)`
4. Consider Celery for production > 50 docs/hour

### Area 4: Text Preprocessing

**Why it matters:** Models work better with cleaned, truncated input; prevents timeouts and errors

**Key recommendations:**
1. Skip summarization for texts < 500 characters
2. Truncate texts > 50,000 characters (use sentence-based truncation)
3. Clean text: remove zero-width chars, fix HTML entities, normalize Unicode
4. Validate summary meets quality thresholds (compression ratio, uniqueness)

### Area 5: Metadata Storage Patterns

**Why it matters:** Tracking summaries enables reproducibility, versioning, and smart regeneration

**Key recommendations:**
1. Store summaries in PostgreSQL JSONB with metadata
2. Track model version with each summary (enables regeneration on upgrade)
3. Keep version history (store old summaries when regenerating)
4. Implement smart regeneration triggers (model upgrade, quality issue, user request)

## Quick Reference: Key Numbers

### Timeouts
- Text summary: 60 seconds (30-120 adaptive)
- Image caption: 30 seconds
- Audio transcription: 300 seconds
- Client-side timeout: 10-20 seconds shorter than server

### Text Lengths
- Minimum for summary: 500 characters
- Optimal range: 1,000-10,000 characters
- Maximum before truncation: 50,000 characters
- DistilBART token limit: 1,024 tokens (≈4,096 characters)

### Performance
- Fast operations: < 30 seconds (use sync)
- Slow operations: > 30 seconds (use async)
- Batch upload: Use async for documents with summaries
- Rate limit: 30-60 requests/minute per client

### Quality Thresholds
- Minimum summary length: 50 characters (10% of original)
- Maximum compression ratio: 0.8 (summary < 20% of original)
- Unique words: > 40% of total words (detect repetition)

## FAQ & Troubleshooting

### Q: Why skip summaries for documents < 500 characters?
A: At this length, summaries are often as long as original text, providing no value. Summary → snippet is better.

### Q: What timeout value should I use?
A: Start with 60 seconds for DistilBART. Use adaptive calculation (see § 3.3) for production.

### Q: Should I show errors when summarization fails?
A: No. Always show fallback (text snippet) instead. Log error server-side for debugging.

### Q: How do I handle users asking "why no summary?"
A: Explain minimum length threshold. If above 500 chars, regenerate and check logs for errors.

### Q: Can I use a different summarization model?
A: Yes, but update config.yml, track model version in metadata, and regenerate old summaries.

### Q: Should summaries be searchable?
A: No. Store in metadata only, not as separate indexed document. Search uses original text.
