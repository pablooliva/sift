### Backup Gap Resolution

`backup.sh` covers items #1-8 but misses #9-11. Two approaches to fix:

- **Option A:** Modify `backup.sh` to include `shared_uploads/`, `logs/frontend/archive/`, and `audit.jsonl`
- **Option B:** Have `cron-backup.sh` wrapper copy them separately before/after `backup.sh` runs

**Recommended: Option A** — these are core data that should be in every backup (manual or automated), not just cron backups. The gap affects interactive `backup.sh` usage too. Adding 3 directory/file copies to `backup.sh` is a minimal, safe change.

**Restore symmetry requirement (Critical Review v2 Finding #1 — IMPORTANT):** `restore.sh` must also be updated to handle these new items. Currently `restore.sh` only knows about `postgres.sql`, `postgres_data/`, `qdrant_storage/`, `txtai_data/`, `document_archive/`, `neo4j.dump`, `neo4j_data/`, `config.yml`, and `.env`. Without a matching restore.sh update, these items would be present in the backup archive but silently ignored during restore — creating a false safety net for irreplaceable data like `shared_uploads/`.

Changes needed in both scripts:
- `backup.sh`: Add backup steps for `shared_uploads/`, `logs/frontend/archive/`, `audit.jsonl` + MANIFEST.txt entries
- `restore.sh`: Add restore steps for the same items + display in backup contents listing

### `shared_uploads/` Permission Model (Critical Review v2 Finding #2 — MEDIUM)

`shared_uploads/` is root-owned (`drwxr-xr-x root root`), created by Docker. Files uploaded through the frontend will be created by the container's user (typically root inside Docker containers).

**Backup direction (reading as `pablo`):** The directory has world-readable permissions (`r-x` for others). Docker's default `umask` (022) means uploaded files should also be world-readable. `cp -r` as `pablo` will succeed for reading.

**Restore direction (writing as `pablo`):** After restore, `shared_uploads/` and its contents will be owned by `pablo:pablo` instead of `root:root`. Docker containers run as root and can read any file regardless of ownership, so this ownership change is functionally harmless — containers will still access the files.

**Edge case — restrictive file permissions:** If a Docker container creates files with `chmod 600 root:root`, `cp -r` as `pablo` would silently skip those files. This is unlikely with default Docker behavior but worth guarding against.

**Mitigation:** After backing up `shared_uploads/`, compare file counts between source and backup. Log a warning if they differ:
```bash
SRC_COUNT=$(find "$PROJECT_ROOT/shared_uploads" -type f | wc -l)
DST_COUNT=$(find "$BACKUP_PATH/shared_uploads" -type f | wc -l)
if [ "$SRC_COUNT" -ne "$DST_COUNT" ]; then
    print_warning "shared_uploads file count mismatch: source=$SRC_COUNT backup=$DST_COUNT (possible permission issue)"
fi
```

### Service Restart Safety (Critical Review Finding #1 — CRITICAL)

`backup.sh` has a dangerous failure mode when used with `--stop`:

1. Line 25: `set -e` — script exits on any error
2. Lines 162-173: Stops `txtai` and `frontend` services
3. Lines 175-295: Backup operations (PostgreSQL dump, directory copies, compression)
4. Lines 321-328: Restarts services — **but only if the script reaches this point**

If ANY unguarded command between lines 173 and 320 fails (disk full during `cp -r`, tar compression error, permission issue), the script exits immediately and **services remain stopped indefinitely**.

Specifically vulnerable commands (NOT wrapped in `if/else`):
- Line 189: `cp -r "$PROJECT_ROOT/postgres_data" "$BACKUP_PATH/postgres_data"`
- Line 201: `cp -r "$PROJECT_ROOT/qdrant_storage" "$BACKUP_PATH/qdrant_storage"`
- Line 213: `cp -r "$PROJECT_ROOT/txtai_data" "$BACKUP_PATH/txtai_data"`
- Line 225: `cp -r "$PROJECT_ROOT/document_archive" "$BACKUP_PATH/document_archive"`
- Line 306: `tar -czf "$ARCHIVE_NAME" "backup_$TIMESTAMP"`

**Resolution:** The cron wrapper MUST guarantee service restart regardless of `backup.sh` exit status. Two-layer defense:

```bash
# In cron-backup.sh:
cleanup() {
    # Always restart services if we're the ones who stopped them
    if [ "$SERVICES_WERE_RUNNING" = true ]; then
        cd "$PROJECT_ROOT"
        docker compose start txtai frontend 2>/dev/null || true
        log "Services restarted after backup (exit code: $?)"
    fi
}
trap cleanup EXIT

# Check if services are running before backup
SERVICES_WERE_RUNNING=false
if docker ps --format '{{.Names}}' | grep -q "^txtai-api$"; then
    SERVICES_WERE_RUNNING=true
fi

# Run backup — trap ensures cleanup on ANY exit
./scripts/backup.sh --stop --output "$BACKUP_EXTERNAL_DIR"
```

This ensures services are restarted whether `backup.sh` succeeds, fails, or is killed.

### Compression Temp-Space (Critical Review Finding — MINOR)

`backup.sh` lines 299-311: When `--output` points to the external drive, compression creates both the uncompressed directory AND compressed archive on the external drive temporarily, then deletes the uncompressed copy. This requires ~2x backup size in temporary space.

With current sizes (~50-100MB compressed) this is negligible. For future reference, if backups grow to multi-GB, this could matter on a nearly-full drive. The minimum free space check should account for 2x the expected backup size.

## External Drive Analysis

### Mount Details

```
Device:     /dev/sdb1 (external USB)
Encryption: LUKS (luks-969b01d5-b215-4842-ade2-344246481d54)
Filesystem: ext4
Mount:      /path/to/external
Manager:    udisks2 (user-space, NOT fstab)
Size:       7.3TB total, 4.7TB available (32% used)
Target:     /path/to/external/backups (already created, empty)
```

### LUKS and Auto-Mount Investigation (Critical Review Finding #5 — IMPORTANT)

**Investigated:** `/etc/crypttab`, `/etc/fstab`, mount manager

**Findings:**
- `/etc/crypttab`: Only contains the internal NVMe OS disk (`nvme0n1p3_crypt`). **No entry for the external 8TB drive.**
- `/etc/fstab`: **No entry for the external drive.** Not configured for auto-mount.
- Mount manager: `udisks2` (user-space, GUI-triggered). Drive appears automatically in file manager after LUKS unlock, but there is no unattended auto-unlock.

**Post-reboot behavior:**
1. System boots → external drive is visible as a locked LUKS device
2. User must manually unlock via GUI (file manager click) or CLI (`udisksctl unlock -b /dev/sdb1`)
3. After unlock, drive auto-mounts to `/path/to/external` via udisks2
4. Until unlocked, `/path/to/external` does not exist as a mount point
5. Cron job detects this via `mountpoint -q` and skips backup with a logged warning

**Could auto-unlock be configured?** Yes, via `/etc/crypttab` with a keyfile, but this defeats LUKS encryption purpose (keyfile on same machine = no protection against physical theft of the whole system). This is a deliberate security trade-off — the user accepts manual unlock after reboot in exchange for actual encryption security.

**Accepted limitation:** After reboot, backups won't run until the user manually unlocks the drive. The cron wrapper logs a clear warning each time it skips, and the staleness detection (see "Failure Notification" below) surfaces this.

### Mount Validation Strategy

```bash
# Check 1: Is actually a mount point (not just a directory on root fs)
# This is THE critical check — prevents writing to root filesystem
mountpoint -q "/path/to/external"

# Check 2: Target directory exists and is writable
[ -d "$BACKUP_EXTERNAL_DIR" ] && [ -w "$BACKUP_EXTERNAL_DIR" ]

# Check 3: Minimum free space (2x expected backup size for compression temp space)
FREE_KB=$(df --output=avail "$BACKUP_EXTERNAL_DIR" | tail -1)
[ "$FREE_KB" -gt "$MIN_FREE_KB" ]
```

If any check fails, the cron job logs a warning and exits without backup — never silently writes to root filesystem.

## Failure Notification (Critical Review Finding #2 — IMPORTANT)

### The Problem

Unattended cron jobs can fail silently. Logging to a file nobody reads is insufficient. The user needs to know when backups aren't happening.

### Available Notification Channels

| Channel | Available? | Suitability |
|---------|-----------|-------------|
| `notify-send` (desktop) | Yes (`/usr/bin/notify-send`) | Good if user has active session |
| `mail` / `sendmail` | No (not installed) | N/A |
| Slack/webhook | Possible but requires setup | Overkill |
| Sentinel file + health check | Always works | Best for staleness detection |

### Chosen Approach: Dual-Layer Notification

**Layer 1: Sentinel file (staleness detection)**

After each successful backup, write a timestamp to a sentinel file:
```bash
# On success:
echo "$(date -Iseconds)" > "$PROJECT_ROOT/logs/backup/last-successful-backup"
```

This file can be checked by:
- The txtai frontend Home page (health check: "Last backup: 2 days ago — WARNING: stale")
- A simple monitoring script
- Manual inspection

**Layer 2: Desktop notification on failure (best-effort) (Critical Review v2 Finding #3 — MEDIUM)**

```bash
# On failure (if DISPLAY is available):
if [ -n "$DISPLAY" ] || [ -n "$WAYLAND_DISPLAY" ]; then
    notify-send --urgency=critical "txtai Backup Failed" "Check logs: $LOG_FILE"
fi
```

**Important: Desktop notification is best-effort, not guaranteed.** It depends on:
- An active GUI session at 3 AM (user may be logged out)
- Correct `DISPLAY` / `DBUS_SESSION_BUS_ADDRESS` values (may change between sessions)
- X11 vs Wayland compatibility (`DISPLAY=:0` is X11-specific; Wayland uses `WAYLAND_DISPLAY`)

Do not over-engineer this layer — the sentinel file (Layer 1) is the reliable mechanism. Desktop notification is a convenience that sometimes works, not a guarantee. The setup script captures current session variables as a best-effort snapshot.

**Layer 3: Failure marker file (for frontend health check)**

On backup failure, write a marker file that the frontend Home page can detect:
```bash
# On failure:
echo "$(date -Iseconds) $ERROR_MSG" > "$PROJECT_ROOT/logs/backup/BACKUP_FAILED"

# On success (clear any previous failure marker):
rm -f "$PROJECT_ROOT/logs/backup/BACKUP_FAILED"
```

This enables future integration with the frontend health check (similar to how `graph.approximate` config validation is displayed on the Home page). The marker file persists until the next successful backup, providing a durable signal that doesn't depend on display sessions.

**Cron environment for notify-send** (captured by setup script, best-effort):
```bash
# In crontab entry:
DISPLAY=:0
DBUS_SESSION_BUS_ADDRESS=unix:path=/run/user/1000/bus
```

### Failure Scenarios Covered

| Scenario | Sentinel File (Layer 1) | Failure Marker (Layer 3) | Desktop Notification (Layer 2) |
|----------|------------------------|-------------------------|-------------------------------|
| Drive not mounted | Stale (no update) | Created | Best-effort, if session active |
| Backup script error | Stale (no update) | Created | Best-effort, if session active |
| Disk full | Stale (no update) | Created | Best-effort, if session active |
| Services left stopped | Stale + services down | Created | Best-effort, if session active |
| Cron not running | Stale (no update) | Stale (no update) | No (cron is the trigger) |
| User not logged in at 3 AM | Stale (checked later) | Persists until next success | No (no display session) |

## Missed-Run Handling (Critical Review Finding #4 — IMPORTANT)

### The Problem

Cron silently skips jobs when the system is off or asleep at the scheduled time. After a reboot, kernel update, or power event at 3 AM, the backup is simply lost.

### Considered Solutions

1. **Systemd timer with `Persistent=true`** — catches up on missed runs, but user explicitly requested cron
2. **anacron** — designed for machines that aren't always on, but adds a dependency
3. **Staleness-based retry in cron wrapper** — check last backup age on every run, backup if stale

### Chosen Approach: Staleness-Based Retry

Add a secondary cron entry that runs at login/boot time (or every few hours) to catch missed backups:

```bash
# Primary: Daily at 3:00 AM
0 3 * * * /path/to/scripts/cron-backup.sh

# Catch-up: Every 6 hours, only runs if last backup is >24h old
0 */6 * * * /path/to/scripts/cron-backup.sh --if-stale 24
```

The `--if-stale HOURS` flag checks the sentinel file timestamp:
- If last successful backup was <HOURS ago → skip (already current)
- If last successful backup was >HOURS ago OR sentinel file missing → run backup
- If sentinel file exists but contains corrupt/unparseable data → treat as stale and run backup
- This covers: reboot recovery, power outage recovery, missed 3 AM window

**Defensive sentinel parsing (Critical Review v2 Finding #5 — LOW):** The sentinel file timestamp must be parsed defensively. A truncated write (power loss during write), filesystem error, or concurrent access could corrupt the file. With `set -e` in the wrapper, a `date` parsing failure would exit the script — meaning a corrupt sentinel file would prevent ALL future backups. Defense:

```bash
if ! last_backup=$(date -d "$(cat "$SENTINEL_FILE" 2>/dev/null)" +%s 2>/dev/null); then
    # Corrupt, empty, or missing — treat as stale (backup needed)
    last_backup=0
fi
```

This provides systemd timer-like catch-up behavior while using cron as requested.

### Drive Not Mounted
- After system reboot (LUKS unlock required — no auto-unlock configured, by design)
- After USB disconnect/reconnect
- After power outage
- **Mitigation**: Mount point validation before backup, staleness-based catch-up after re-mount

### Disk Space Exhaustion
- Without retention policy: unlimited growth
- Compressed backup size: ~50-100MB initially (empty databases + Neo4j overhead), growing with data
- 4.7TB available = effectively unlimited for this use case
- **Mitigation**: Configurable retention policy + minimum free space check (2x expected backup size)

### Docker Not Running
- Services might be stopped for maintenance
- `backup.sh` already handles this (falls back to directory copies)

### Cron Environment Differences
- `PATH` may not include docker — verified: `docker` is at `/usr/bin/docker`, `docker compose` works with minimal `PATH=/usr/bin:/bin`
- Environment variables from `.env` not automatically loaded — must use `set -a` before sourcing (`.env` has no `export` prefixes)
- `HOME` and working directory may differ
- `DISPLAY` and `DBUS_SESSION_BUS_ADDRESS` needed for `notify-send`
- **Mitigation**: Explicit PATH in crontab, `set -a; source .env; set +a` pattern, capture display variables in setup

### Concurrent Backup Runs
- Staleness catch-up cron entry could overlap with primary 3 AM entry
- If backup takes >6 hours, catch-up could fire during primary run
- **Mitigation**: Lock file (`flock`) prevents concurrent runs — second invocation skips immediately
- **Lock file location (Critical Review v2 Finding #4 — LOW):** Must be on the local filesystem, NOT on the external drive (which may not be mounted). Location: `$PROJECT_ROOT/logs/backup/.cron-backup.lock`. This ensures `flock` always works regardless of external drive state. The lock file is acquired before the mount point check, so even if the drive is unmounted, concurrent instances are still prevented.

### Permission Issues
- Cron runs as `pablo` (same user, `docker` group membership preserved)
- External drive owned by `pablo` (confirmed: `drwxrwxr-x pablo pablo`)
- Should not be an issue, but worth logging

### Backup Script Fails Mid-Way (services stopped)
- `backup.sh --stop` stops services, then `set -e` could exit before restart
- **Mitigation**: Cron wrapper's `trap cleanup EXIT` ensures service restart on ANY exit — see "Service Restart Safety" above

## Design Decisions

### Approach: Cron Wrapper Around Existing backup.sh

**Chosen**: Create a `cron-backup.sh` wrapper that:
1. Rotates log file if >10MB (atomic `mv`)
2. Acquires lock (`flock` on `$PROJECT_ROOT/logs/backup/.cron-backup.lock`)
3. Checks staleness with defensive parsing (skip if `--if-stale` flag and recent backup exists; corrupt sentinel → treat as stale)
4. Validates mount point (`mountpoint -q`)
5. Sources environment (`set -a; source .env; set +a`)
6. Sets up `trap cleanup EXIT` (guarantees service restart)
7. Calls `backup.sh --stop --output "$BACKUP_EXTERNAL_DIR"`
8. Updates sentinel file on success / creates `BACKUP_FAILED` marker on failure
9. Applies retention policy (delete old backups)
10. Logs results
11. Sends desktop notification on failure (best-effort)

**Why not modify backup.sh directly?**
- `backup.sh` is already production-stable and well-tested
- Interactive use should remain unchanged
- Cron concerns (logging, mount validation, lock files, traps, notifications) don't belong in general-purpose backup tool
- Separation of concerns: backup logic vs. scheduling logic

### Cron vs Systemd Timer

| Aspect | Cron | Systemd Timer |
|--------|------|---------------|
| Setup complexity | Simple (one line) | More complex (2 files: service + timer) |
| Logging | Manual (redirect to file) | Automatic (journald) |
| Dependency management | None | Can declare dependencies |
| User familiarity | Universal | Linux-specific |
| Missed runs | Skipped (mitigated by staleness catch-up) | Can catch up (Persistent=true) |
| Monitoring | Sentinel file + notify-send | `systemctl status`, journalctl |

**Decision**: Use **cron** with staleness-based catch-up. The user explicitly requested cron. The staleness catch-up mechanism provides equivalent missed-run recovery to systemd's `Persistent=true`. The only remaining gap is if cron itself isn't running (unlikely on a systemd-managed system where cron is enabled).

### Retention Policy Options

| Strategy | Pros | Cons |
|----------|------|------|
| Keep last N backups | Simple, predictable space | Loses history beyond N days |
| Keep last N days | Time-based, intuitive | Space varies |
| Grandfather-father-son | Good coverage | Complex to implement |
| Keep all | No data loss | Unbounded growth |

**Recommendation**: Keep last **30 days** by default, configurable via `BACKUP_RETENTION_DAYS` ENV variable. Compressed backups are ~50-100MB initially, growing with data. 30 days ≈ 1.5-3GB initially — negligible on 8TB drive. Users can set to 0 to keep all.

**Retention execution order**: Always run backup FIRST, then apply retention cleanup. This ensures a new backup exists before deleting old ones.

### Services: Stop or Leave Running?

For 3:00 AM automated backups, **stopping services** (`--stop`) provides consistency guarantees:
- No writes during PostgreSQL dump
- No partial vector updates in Qdrant
- Brief downtime (~2-5 minutes) at 3 AM is acceptable

**Decision**: Use `--stop` by default for cron backups (consistency over availability at 3 AM). The trap handler guarantees restart even on failure.

### Log Management

Cron wrapper logs to a dedicated file:
- Location: `$PROJECT_ROOT/logs/backup/cron-backup.log`
- Include timestamps, backup size, duration, success/failure, skip reasons
- Sentinel file: `$PROJECT_ROOT/logs/backup/last-successful-backup` (ISO 8601 timestamp)
- Failure marker: `$PROJECT_ROOT/logs/backup/BACKUP_FAILED` (created on failure, removed on success)

**Log rotation (Critical Review v2 Finding #6 — LOW):** Rotate atomically by moving the current log to a `.1` suffix, rather than truncating in-place. In-place truncation while the file could be appended to (if `flock` fails for any reason) risks data corruption. Rotation approach:

```bash
# At start of each run, check log size
if [ -f "$LOG_FILE" ] && [ "$(stat -c%s "$LOG_FILE" 2>/dev/null || echo 0)" -gt 10485760 ]; then
    mv "$LOG_FILE" "${LOG_FILE}.1"  # Atomic rename, old .1 is overwritten
fi
```

This keeps at most 2 log files (~20MB max). The `flock` already prevents concurrent runs, but atomic rotation is safer than in-place truncation as defense-in-depth.
